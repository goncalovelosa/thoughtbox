# Code Mode Reference Repos

## What Code Mode does

A standard MCP server exposes every capability as a separate tool. An agent calling the Cloudflare API through native MCP would see ~2,500 individual tools, consuming ~244k tokens of context just for the tool schemas. Code Mode collapses all of that into 2 tools and ~1k tokens.

The trick: instead of the agent picking a tool and filling in parameters, the agent writes a short TypeScript/JavaScript function that calls a typed SDK. That function runs in a sandboxed executor on the server side. The agent gets back the return value.

This gives the agent three things it doesn't get from traditional tool-per-endpoint MCP:

1. **Composability** -- chain multiple API calls, branch on intermediate results, and do logic in a single round-trip instead of one tool call per operation.
2. **Token efficiency** -- the full API surface is described by a typed SDK interface (a few hundred tokens), not thousands of individual tool schemas.
3. **Security boundary** -- auth tokens and secrets live in the server's `request()` function. The sandbox that runs agent-generated code has no outbound network access and never sees credentials.

## The two-tool pattern

Every Code Mode server exposes the same pair:

| Tool | What the agent does with it |
| --- | --- |
| `search` / `search_docs` | Query the API spec or SDK docs to discover endpoints, parameters, and schemas. |
| `execute` / `code` | Write a function against the typed SDK. The server executes it in a sandbox and returns the result. |

The agent's workflow: search first to find the right endpoints, then execute code against them. Both steps happen through the same MCP connection.

## Why this matters for Thoughtbox

Thoughtbox's MCP server currently has a parameter-heavy gateway tool (`thoughtbox_gateway`) where every operation requires the agent to construct the right combination of `operation`, `args`, and nested fields. Code Mode would replace that with a TypeScript API where the agent writes:

```ts
async () => {
  const session = await tb.startNew({ title: "Research session" });
  await tb.thought({ sessionId: session.id, content: "First observation", thoughtType: "observation" });
  const state = await tb.getState();
  return state;
}
```

Instead of three separate tool calls with complex parameter objects, the agent composes the full workflow in one execution.

## Key implementation details across repos

**Executor isolation**: Cloudflare uses `DynamicWorkerExecutor` (Worker-based sandbox). Dodo Payments uses either a Deno subprocess with restricted permissions (`--allow-read`, `--allow-net=<api-host-only>`) or a remote Stainless sandbox. Both prevent the agent's code from accessing the network or filesystem directly.

**Auth stays on the host**: In every implementation, the server-side `request()` function injects auth headers. The sandbox environment exposes a `cloudflare.request()` or `client.*` interface that proxies through the host. Agent-generated code never touches tokens.

**Spec delivery**: Cloudflare pre-processes and caches the OpenAPI spec in R2, serving it to the `search` tool. Dodo Payments fetches SDK documentation from a Stainless-hosted search endpoint. Both approaches keep the full API surface off the agent's context window.
