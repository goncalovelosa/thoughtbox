Assuming you mean Model Context Protocol, my bet is: the best underbuilt use case is an MCP-native exception-resolution server for messy, approval-heavy operations.

Today’s MCP ecosystem is still dominated by connectors. Official MCP docs use file systems, databases, GitHub, Slack, and calendars as the canonical examples, and large community directories are crowded with browser automation, communication, database, file system, and memory servers.  ￼

What makes exception-resolution different is that MCP already has the right primitives for it. MCP is stateful; servers expose resources, prompts, and tools; servers can request sampling and elicitation; authorization is standardized for sensitive operations; and the Tasks SEP adds call-now/fetch-later execution for long-running workflows. The protocol also includes notifications/progress patterns for live updates. That is much closer to a governed workflow engine than to a simple plugin API.  ￼

So the killer app is not “let the model query Postgres” or “let the model click around Slack.” It is: “here is a live case, here is the evidence, here is the policy, here are the reversible actions, here are the approval gates, and here is the durable state until the case is closed.” Think failed renewals, invoice disputes, vendor exceptions, insurance claims, CAPA/deviation handling, support escalations, or incident response. The model stops being a summarizer and becomes a bounded operator inside a queue.  ￼

I can’t prove nobody has done it, but the few orchestration/task projects I found are still early and mostly dev-oriented—for example, task orchestrators pitched as memory/coordination layers for AI-assisted development, with limited open-source footprints, and one explicitly warns it is for development/experimentation rather than production or critical systems.  ￼

If I were building a company on this, I’d start with order-to-cash exception autopilot. Adjacent MCP products are already appearing around ERP analytics and quote-to-cash context, so the data plane is starting to exist; the missing layer is a server that owns case state, resolution proposals, approval routing, and final execution. MVP shape: resources/cases/*, resources/policies/*, prompts/propose_resolution, prompts/request_approval, tools/simulate, tools/execute, tools/escalate, plus tasks/* and subscriptions until closure.  ￼

In one sentence: the biggest unclaimed MCP opportunity is a policy-aware workflow state machine for exception queues, not another thin wrapper around existing SaaS APIs.
